Sistema de Layout

app.blade.php (Layout Tenant)

Layout principal para todas as páginas do tenant.

  • @yield('title') — Título da página
  • @yield('content') — Conteúdo principal
  • @stack('styles') — CSS adicional
  • @stack('scripts') — JS adicional
  • Inclui: AdminLTE sidebar, navbar, Vite assets, Toastr, Echo config
  • Reverb config: injetado via window.reverbConfig no Blade (não usa VITE_*)
master.blade.php (Layout Master)

Layout para o painel do super admin.

  • Similar ao app.blade.php mas com sidebar/navbar diferentes
  • Menu com links para tenants, planos, logs, sistema

Padrão Viewer Disable

Filosofia: Nunca ESCONDER botões de viewers — apenas DESABILITAR. O viewer deve ver a interface completa mas sem poder interagir com ações de escrita.
Padrão Blade (server-side)

Adicionar o atributo disabled diretamente no HTML renderizado:

<!-- Botão com disabled condicional -->
<button class="btn-primary-sm" {{ auth()->user()->isViewer() ? 'disabled' : '' }}>
    Novo Lead
</button>

<!-- Botão com disabled + estilo de opacity -->
<button class="btn-primary-sm"
    {{ auth()->user()->isViewer() ? 'disabled style=opacity:.5;pointer-events:none;' : '' }}>
    <i class="bi bi-plus-lg"></i> Novo Lead
</button>
Padrão JS (client-side)

Usar window.isViewer para condicionar features interativas:

// Desabilitar drag-drop no Kanban
if (!window.isViewer) {
    new Sortable(columnEl, { ... });
}

// Chat: substituir input por mensagem
if (window.isViewer) {
    inputArea.innerHTML = 'Modo visualização';
}
Exemplos no projeto
  • Kanban: drag-drop desabilitado para viewers
  • Contatos: botões "Novo Lead", "Importar" e FAB ficam com disabled + opacity:.5
  • WhatsApp: input de chat substituído por texto "Modo visualização"
  • Settings: formulários e botões de salvar desabilitados

Sidebar

Comportamento de Colapso

A sidebar colapsa por padrão (72px). O estado é persistido em localStorage com a chave sidebar_collapsed.

Anti-flash Pattern

Para evitar o flash visual (sidebar expandida que colapsa depois do JS carregar), o estado é lido antes do primeiro render com inline scripts no <head>:

<!-- No <head>, antes de qualquer render -->
<script>
    var s = localStorage.getItem('sidebar_collapsed');
    window.__sidebarCollapsed = s === null ? true : s === '1';
</script>

<!-- Logo após abrir a tag do sidebar -->
<aside class="sidebar" id="sidebar">
<script>if(window.__sidebarCollapsed) document.getElementById('sidebar').classList.add('sidebar--collapsed');</script>

<!-- Idem para topbar e main-content (ajustam margin-left) -->
<script>if(window.__sidebarCollapsed && window.innerWidth > 768) document.getElementById('topbar').style.left='72px';</script>
<script>if(window.__sidebarCollapsed && window.innerWidth > 768) document.getElementById('mainContent').style.marginLeft='72px';</script>
Transições: Sidebar, topbar e main-content iniciam com style="transition:none;". Após o primeiro frame, um requestAnimationFrame restaura as transições CSS para que toggles subsequentes sejam animados.
Estrutura Interna
  • Logo + collapse btn: fixo no topo (.sidebar-logo, flex-shrink:0)
  • Nav items: dentro de .sidebar-nav-scroll — container scrollável (flex:1; overflow-y:auto;) com scrollbar oculta
  • User footer: fixo na parte inferior (.user-card)
Mobile
  • Sidebar fica off-canvas (transform: translateX(-100%)) e abre com classe .open
  • Mesmo se sidebar--collapsed no desktop, no mobile os estilos são sobrescritos via media query para exibir a sidebar sempre expandida (260px)
  • Labels, ícones, submenu e user info são todos forçados para visíveis com !important

Trial Widget

Lógica de Exibição

Exibido apenas para tenants com status = 'trial' e trial_ends_at no futuro. Computado no layout:

// Variáveis computadas no layout
$__showTrial = $__tenant && $__tenant->status === 'trial'
    && $__tenant->trial_ends_at && !$__tenant->trial_ends_at->isPast();
$__trialDays  = now()->diffInDays($__tenant->trial_ends_at, false);
$__trialTotal = $__tenant->created_at->diffInDays($__tenant->trial_ends_at);
$__trialPct   = max(0, min(100, ($__trialDays / $__trialTotal) * 100));
Desktop: .trial-widget

Posicionado dentro da topbar. Exibe texto "Trial: X dias" com ícone de relógio e uma barra de progresso animada (barber pole).

  • Cor da barra: #F97316 (laranja)
  • Animação: listras diagonais via linear-gradient com @keyframes barber pole
  • Escondido no mobile via display: none !important na media query max-width: 768px
Mobile: .trial-mobile-banner

Banner no topo do .main-content, visível apenas em telas menores que 768px.

  • Fundo: #FFEDD5

resources/views/errors/403.blade.php

Página standalone (sem layout do tenant) para erros de autorização.

  • Layout: centrado na tela com display:flex; align-items:center; justify-content:center;
  • Logo: logotipo da plataforma no topo
  • Ícone: bi-shield-lock em círculo vermelho (#fef2f2 fundo, #ef4444 ícone)
  • Código: "403" em font-size:96px; color:#e5e7eb;

No perfil do lead, o botão "Editar" exibe apenas o ícone de lápis no mobile e o texto completo no desktop:

Blade Gotchas

{{ }} dentro de <script>

Blade interpreta {{ }} como <?= ?> mesmo dentro de tags <script>, incluindo em strings e comentários JS.

NUNCA usar double curly braces em JS, nem em comentários. Use String.fromCharCode(123,123) se precisar da string literal.

@php inline vs bloco

@php($var = value) em child views compila para <?php($var = value) SEM fechamento ?>. O bloco PHP fica aberto e toda a view falha.

SEMPRE usar bloco: 
@php
    $title = 'CRM';
    $pageIcon = 'fas fa-columns';
@endphp

@json() com closures

@json() usa regex single-line e falha com closures ou arrays multi-linha complexos.

Usar: @php ... @endphp + {!! json_encode($var) !!}
