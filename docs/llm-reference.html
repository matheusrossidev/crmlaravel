LLM Reference

Referencia rapida para LLMs e assistentes de IA entenderem e modificarem o codigo

Esta pagina e para LLMs. Contem resumos, padroes, gotchas, tabelas de permissao e quick-references otimizados para AI assistants que precisam entender e modificar o codigo do Syncro CRM.

Resumo Executivo

Syncro e uma plataforma SaaS multi-tenant de CRM, automacao e comunicacao omnichannel. Construida com Laravel 11 + PHP 8.2/8.3, MySQL, AdminLTE 4, Bootstrap 5, jQuery. Deploy via Docker Swarm no dominio app.syncro.chat.

Integra WhatsApp (WAHA), Instagram (Meta Graph API), AI agents (OpenAI/Agno), pagamentos (Asaas), e tem um chatbot visual builder com widget para websites.

Conceitos-chave:

  • Multi-tenancy via BelongsToTenant trait (Global Scope automatico)
  • Role-based permissions com 3 roles: admin manager viewer
  • Pipeline filtering via pipeline_user pivot — manager/viewer podem ver apenas pipelines atribuidas
  • Department filtering — conversas filtradas por departamento para manager/viewer
  • RoleMiddleware controla acesso a rotas; UI usa disable (nao hide) para viewers

RoleMiddleware — Como Funciona

// app/Http/Middleware/RoleMiddleware.php
class RoleMiddleware
{
    public function handle(Request $request, Closure $next, string ...$roles): Response
    {
        $user = $request->user();

        // Super admin sempre passa
        if ($user->isSuperAdmin()) return $next($request);

        if (! in_array($user->role, $roles, true)) {
            abort(403, 'Sem permissao para esta acao.');
        }

        return $next($request);
    }
}

Uso em Rotas

// Admin only:
Route::middleware('role:admin')->group(function () {
    Route::post('pipelines', [PipelineController::class, 'store']);
});

// Admin + Manager:
Route::middleware('role:admin,manager')->group(function () {
    Route::post('/contatos', [LeadController::class, 'store']);
    Route::put('/contatos/{lead}', [LeadController::class, 'update']);
});

// Sem middleware = todos os roles (admin, manager, viewer) podem acessar
Route::get('/crm', [KanbanController::class, 'index']);

Pipeline Filtering — allowedPipelineIds()

[Request chega] | v [auth()->user()->allowedPipelineIds()] | +---+---+ | | null [1,3] | | v v Sem filtro whereIn('pipeline_id', [1,3]) (admin) (manager/viewer com pipelines atribuidas) 
// User model:
public function allowedPipelineIds(): ?array
{
    if ($this->isAdmin()) return null;  // admin ve tudo

    $ids = $this->pipelines()->pluck('pipelines.id')->toArray();
    return count($ids) > 0 ? $ids : null;  // sem atribuicao = sem restricao
}

// Uso no controller:
$pipelineIds = auth()->user()->allowedPipelineIds();

$leads = Lead::query()
    ->when($pipelineIds, fn($q) => $q->whereIn('pipeline_id', $pipelineIds))
    ->get();

Department Filtering

Admin ve TODAS as conversas. Manager/viewer so veem conversas cujo departamento pertence aos departamentos do usuario (via department_user pivot). O campo can_see_all_conversations no User permite override.

Blade Gotchas Criticos

1. {{ }} dentro de <script>

Blade interpreta {{ }} como PHP echo em QUALQUER lugar, incluindo strings JS e comentarios.

// ERRADO — Blade vai quebrar:
// Convert {{var}} to HTML
var x = "{{nome}}";

// CORRETO:
// Convert variable placeholders to HTML
var x = String.fromCharCode(123,123) + 'nome' + String.fromCharCode(125,125);

2. @php inline em child views

@php($var = value) compila para <?php($var = value) SEM ?>. O bloco PHP fica aberto e toda a view falha.

// ERRADO:
@php($title = 'CRM')

// CORRETO:
@php
    $title = 'CRM';
@endphp

3. @json() com arrays complexos

Usar @php + {!! json_encode() !!} em vez de @json() para arrays multi-linha.

4. VITE_* env vars em producao

Dockerfile roda npm run build SEM build args. import.meta.env.VITE_* sera undefined em producao. Usar window.* injetado via Blade.

5. Sidebar anti-flash pattern

O layout injeta um <script> inline no <head> que le localStorage('sidebar_collapsed') e aplica a classe ANTES do render. Isso evita o "flash" da sidebar aberta antes do JS carregar.

<!-- No <head> do layout, ANTES de qualquer CSS transicional -->
<script>
(function(){
    var s = localStorage.getItem('sidebar_collapsed');
    window.__sidebarCollapsed = s === null ? true : s === '1';
}());
</script>

<!-- No mainContent, aplica margin imediatamente: -->
<script>
if(window.__sidebarCollapsed && window.innerWidth > 768)
    document.getElementById('mainContent').style.marginLeft = '72px';
</script>

6. Trial widget

O layout computa $__showTrial (tenant em trial com data futura). Dois widgets: desktop (texto + barber pole bar no topbar) e mobile (banner no main-content).

// Calculado no layout:
$__showTrial = $__tenant
    && $__tenant->status === 'trial'
    && $__tenant->trial_ends_at
    && !$__tenant->trial_ends_at->isPast();

$__trialDays = $__showTrial
    ? (int) now()->diffInDays($__tenant->trial_ends_at, false)
    : 0;

Padroes de Codigo

Controller Pattern (basico)

// Padrao: Controller leve, Service para logica complexa
class MeuController extends Controller
{
    public function store(Request $request): JsonResponse
    {
        $request->validate(['name' => 'required|string|max:191']);

        $item = MeuModel::create([
            'tenant_id' => auth()->user()->tenant_id,
            'name' => $request->name,
        ]);

        return response()->json(['success' => true, 'data' => $item]);
    }
}

Role Middleware em Rotas

// Padrao usado em routes/web.php:
// 1. Rotas de LEITURA fora do middleware (todos podem ver)
// 2. Rotas de ESCRITA dentro de middleware('role:...')

Route::prefix('agenda')->name('calendar.')->group(function () {
    // Leitura — todos os roles
    Route::get('/',        [CalendarController::class, 'index']);
    Route::get('/eventos', [CalendarController::class, 'events']);

    // Escrita — admin + manager
    Route::middleware('role:admin,manager')->group(function () {
        Route::post('/eventos',        [CalendarController::class, 'store']);
        Route::put('/eventos/{id}',    [CalendarController::class, 'update']);
        Route::delete('/eventos/{id}', [CalendarController::class, 'destroy']);
    });
});

Pipeline Filtering em Controller

// Padrao usado em KanbanController, DashboardController, LeadController:
public function index()
{
    $user = auth()->user();
    $pipelineIds = $user->allowedPipelineIds(); // null ou int[]

    $pipelines = Pipeline::query()
        ->when($pipelineIds, fn($q) => $q->whereIn('id', $pipelineIds))
        ->with('stages')
        ->get();

    $leads = Lead::query()
        ->when($pipelineIds, fn($q) => $q->whereIn('pipeline_id', $pipelineIds))
        ->get();

    return view('tenant.crm.kanban', compact('pipelines', 'leads'));
}

Webhook Processing Pattern

// 1. Controller recebe e sempre retorna 200
// 2. Job processa (sync ou queued)
// 3. Dedup via Cache::add() ou UNIQUE constraint
// 4. Excecoes logadas, nunca propagadas ao caller

public function handle(Request $request): Response
{
    try {
        ProcessWahaWebhook::dispatchSync($request->all());
    } catch (\Throwable $e) {
        Log::channel('whatsapp')->error($e->getMessage());
    }
    return response('OK', 200);
}

Decisoes Arquiteturais

DecisaoPor que
Processamento sincrono de webhooks WAHAGarantir ordem de processamento e dedup efetiva. Queues causariam race conditions piores.
BelongsToTenant via Global ScopeIsolamento automatico sem precisar de WHERE manual em cada query.
jQuery em vez de React (maioria)Projeto iniciou com AdminLTE (jQuery-based). React usado apenas no chatbot builder.
Blade + inline JS em vez de SPASimplicidade. Cada pagina e server-rendered com JS inline para interatividade.
Widget como script publico (vanilla JS)Deve funcionar em qualquer site sem dependencias. Zero build tools necessarios pelo cliente.
Reverb em vez de PusherSelf-hosted, sem custo por conexao. Roda como service no Docker Swarm.
Steps JSON no ChatbotFlowFluxo serializado como JSON permite builder visual simples e execution engine direta.
Agno como microservicoSeparacao de concerns. LLM processing pode escalar independentemente.
RoleMiddleware (nao Gates/Policies)Simplicidade: 3 roles fixos, sem permissoes granulares. Middleware em grupo de rotas e mais claro que Policy checks espalhados.
Disable vs Hide para viewerViewer precisa entender o que existe na plataforma. Esconder botoes causa confusao. Desabilitar e mais user-friendly.
Pipeline filtering via pivotFlexivel: admin sempre ve tudo; manager/viewer podem ser restritos a pipelines especificos sem mudar logica de tenant.
