Multi-Tenancy

Padrão: Cada tenant (empresa) é isolado por tenant_id em todas as tabelas. O isolamento é automático via Global Scope.

Como funciona

// app/Models/Traits/BelongsToTenant.php
trait BelongsToTenant {
    protected static function booted() {
        // Adiciona WHERE tenant_id = ? em TODAS as queries
        static::addGlobalScope('tenant', function($query) {
            $tenantId = auth()->user()?->tenant_id
                     ?? request()->attributes->get('tenant_id');
            if ($tenantId) $query->where('tenant_id', $tenantId);
        });

        // Auto-preenche tenant_id ao criar
        static::creating(function($model) {
            if (!$model->tenant_id) {
                $model->tenant_id = auth()->user()?->tenant_id
                    ?? request()->attributes->get('tenant_id');
            }
        });
    }
}

RoleMiddleware — Uso

Sintaxe: ->middleware('role:admin') para uma role ou ->middleware('role:admin,manager') para várias. Super admins (is_super_admin = true) sempre passam sem verificação. 
// app/Http/Middleware/RoleMiddleware.php
class RoleMiddleware
{
    public function handle(Request $request, Closure $next, string ...$roles): Response
    {
        $user = $request->user();

        // Super admin sempre passa
        if ($user->isSuperAdmin()) {
            return $next($request);
        }

        if (! in_array($user->role, $roles, true)) {
            abort(403, 'Sem permissão para esta ação.');
        }

        return $next($request);
    }
}

Resumo de Roles

  • viewer — Somente leitura. Pode consultar dashboard, kanban, contatos, conversas (filtradas por departamento) e relatórios. Não pode criar, editar ou excluir nenhum recurso.
  • manager — Operações de CRM. Tudo do viewer + criar/editar leads, enviar mensagens, gerenciar campanhas, criar eventos. Sem acesso a configurações ou integrações de escrita.
  • admin — Acesso completo ao tenant. Tudo do manager + configurações, integrações, chatbot, agentes IA, gerenciamento de usuários e billing.

Padrão de UI: Desabilitar, não Esconder

Convenção: Para ações restritas, o botão/link deve ficar visível mas desabilitado (disabled, com tooltip explicativo) em vez de ser escondido. Isso permite ao usuário saber que a funcionalidade existe mesmo sem ter permissão.

Isolamento por Pipeline

Conceito: Além do isolamento por tenant, usuários manager e viewer podem ter acesso restrito a pipelines específicos via a tabela pivot pipeline_user.

Como funciona

O model User possui a relação pipelines() (BelongsToMany) e o helper allowedPipelineIds():

// app/Models/User.php
public function pipelines(): BelongsToMany
{
    return $this->belongsToMany(Pipeline::class);
}

/**
 * Retorna IDs das pipelines permitidas, ou null se sem restrição.
 * Admin sempre vê tudo. Se nenhuma pipeline atribuída = sem restrição.
 */
public function allowedPipelineIds(): ?array
{
    if ($this->isAdmin()) {
        return null;  // admin vê tudo
    }

    $ids = $this->pipelines()->pluck('pipelines.id')->toArray();

    return count($ids) > 0 ? $ids : null;  // sem atribuição = sem restrição
}

Fluxo de Decisão

User ──> allowedPipelineIds() │ isAdmin()? ──sim──> return null ──> vê TODOS os pipelines │ não pipeline_user tem registros? ──não──> return null ──> vê TODOS (retrocompat.) │ sim return [id, id, ...] ──> filtra por whereIn('pipeline_id', $ids)

Padrão de uso nos Controllers

// Padrão aplicado em KanbanController, LeadController, DashboardController
$allowedPipelineIds = auth()->user()->allowedPipelineIds();

$leads = Lead::query()
    ->when($allowedPipelineIds, fn($q) => $q->whereIn('pipeline_id', $allowedPipelineIds))
    ->get();

Onde é aplicado

ControllerMétodo(s)O que filtra
KanbanControllerindex, pollLista de pipelines, leads no kanban
LeadControllerindex, show, editLista de contatos, pipelines no formulário
DashboardControllerindex, buildDashboardDataMétricas, gráficos, funil por pipeline
Retrocompatível: Se um usuário manager/viewer não tiver nenhuma entrada na tabela pipeline_user, o método retorna null e nenhum filtro é aplicado — o usuário vê todos os pipelines do tenant. Isso garante que tenants existentes não precisem configurar a tabela pivot para continuar funcionando.

Filtragem de Conversas por Departamento

Conceito: Conversas de WhatsApp e Instagram são filtradas por departamento para usuários não-admin. Isso garante que cada equipe veja apenas as conversas relevantes ao seu setor.

Regras de Visibilidade

RoleConversas visíveiscan_see_all_conversations
admin Todas as conversas do tenant, independente de departamento Pode ser true (mas admin já vê tudo pela role)
manager Conversas do(s) departamento(s) do usuário + sem departamento + atribuídas ao usuário Sempre false (campo existe mas só admin pode ter true)
viewer Conversas do(s) departamento(s) do usuário + sem departamento + atribuídas ao usuário Sempre false

Lógica de Filtragem

// WhatsappController — aplicado em index() e poll()
$authUser = auth()->user();
$restrictByDept = !$authUser->isAdmin();
$userDeptIds = $restrictByDept
    ? $authUser->departments()->pluck('departments.id')
    : collect();

if ($restrictByDept) {
    $query->where(function ($q) use ($authUser, $userDeptIds) {
        $q->whereIn('department_id', $userDeptIds)    // meu departamento
          ->orWhereNull('department_id')               // sem departamento
          ->orWhere('assigned_user_id', $authUser->id); // atribuída a mim
    });
}

Fluxo de Decisão

User abre /chats │ isAdmin()? ──sim──> vê TODAS as conversas │ não Carrega departments() do user │ Filtra conversas: dept_id IN (meus deptos) OR dept_id IS NULL OR assigned_user_id = meu id

Onde é aplicado

ControllerMétodo(s)Canais filtrados
WhatsappControllerindex, pollWhatsApp + Instagram (ambos filtrados no mesmo controller)
Nota: O campo can_see_all_conversations existe na tabela users mas, na implementação atual, a filtragem é baseada exclusivamente em isAdmin(). Apenas admins podem ter esse campo como true. Managers e viewers são sempre filtrados por departamento.
